# RAG Metrics Refinement Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Preserve the historical retrieval baselines for retrospective analysis while moving future retrieval refinement around the current eval-approved candidate: `chunk_dense` with Cloudflare Workers AI `@cf/baai/bge-m3`, without pretending retrieval scores prove final answer quality.

**Architecture:** Keep production `answer-question` unchanged while iterating offline through `rag_eval_*` and `rag_retrieval_*`. The next retrieval lever is not an embedding swap; it is chunk-level retrieval, entity-aware query expansion, fusion, and reranking. Generation quality is a separate L2 gate that must be evaluated before claiming full RAG quality or changing production answer behavior.

**Tech Stack:** Supabase PostgreSQL, pgvector, pg_trgm/BM25-style lexical retrieval, PostgREST RPC, Node 20 CLI scripts, `@cf/baai/bge-m3` embeddings for the refinement/eval path, Cloudflare Workers AI `@cf/baai/bge-reranker-base` as the preferred reranker candidate, TokenRouter JSON calls for query rewrite/LLM-judge fallback when needed, existing `rag_eval_*`, `rag_retrieval_*`, and `article_chunks` tables.

Embedding direction: the refinement target is not pure vector retrieval. Use `@cf/baai/bge-m3` as the dense vector model inside a BM25/lexical + vector hybrid retriever. Do not mix BGE query vectors with existing Cohere-generated document vectors. Any BGE replay must use BGE query embeddings and BGE document/chunk embeddings generated by the same model. BGE-M3 can support dense, sparse, and multi-vector retrieval modes, but this plan's current implementation uses dense BGE vectors plus Postgres lexical/BM25-style recall; true BGE sparse or multi-vector storage should be added only after confirming the chosen Workers AI/API response exposes those weights in a usable form.

## Retrieval Metrics Are Not Generation Metrics

The current `chunk_dense` numbers are retrieval-only scores. They mean the model is more likely to receive useful reference material; they do **not** prove the final generated answer is correct, relevant, or faithful to the context.

Before any production answer-quality claim, add a generation-side evaluation pass over the same `qa-v1-2026-06` cases:

| Layer | Metric | What It Answers | Gate Before Claiming Full RAG Quality |
|---|---|---|---|
| Retrieval | Recall@K, MRR, NDCG, Hit@5 | Did we retrieve and rank the right evidence? | Already measured for `chunk_dense`. |
| Context | Context precision / context recall | Was the injected context useful and complete? | Add after chunk coverage is stable. |
| Generation | Faithfulness / groundedness | Did the answer stay inside the provided context? | Required before production answer-quality claims. |
| Generation | Answer relevancy | Did the answer actually answer the question? | Required before production answer-quality claims. |
| Business | thumbs-down rate, follow-up rate, badcase rate | Did users actually find the answer useful? | Required after production rollout. |

Recommended tooling: implement a lightweight local evaluator first, then optionally compare against RAGAS or DeepEval-style metrics. The evaluator should store every generated answer, injected context, judge model, judge prompt version, faithfulness score, answer-relevancy score, and human override in `rag_eval_*` tables or a separate generation-eval table. Do not overwrite retrieval baselines when adding generation metrics.

Resume-safe wording until this exists: "improved offline retrieval metrics." Avoid "improved RAG answer accuracy" until faithfulness and answer relevancy are measured.

## Agentic RAG Architecture Layer

Agentic RAG is the next architecture level for this project, but it should be a gated orchestration layer above the existing retriever, not a replacement for the fast baseline path.

The current production `answer-question` path is a linear RAG pipeline: route, retrieve, generate. The proposed agentic path adds planning, tool selection, context critique, and bounded re-retrieval only when the query needs it.

Target routing:

| Query Type | Path | Reason |
|---|---|---|
| Simple article question | Existing linear RAG | Lowest latency and enough context. |
| Entity-heavy or exact-event question | Chunk dense + lexical/entity hybrid | Better named-entity and event recall. |
| Ambiguous multi-turn question | Query completion + retrieval | Resolves pronouns and missing context. |
| Comparison / multi-hop question | Agentic RAG | Needs decomposition and multiple retrieval calls. |
| Low-context or conflicting evidence | Critique agent then re-retrieve | Avoids confident hallucination. |

Agentic RAG components:

1. **Intent Router**
   - Classifies the query as simple, entity-heavy, multi-hop, ambiguous, or answerability-risky.
   - Defaults to the existing fast path unless the query clearly benefits from agentic work.

2. **Planner**
   - Converts complex questions into one to three retrieval subgoals.
   - Examples: compare two companies, find before/after timeline, connect source/entity/event.
   - Must emit a small JSON plan with `subqueries`, `required_evidence`, and `stop_condition`.

3. **Query Rewrite Layer**
   - Supports entity expansion, HyDE-style semantic query, task decomposition, and conversation context completion.
   - Uses drift guardrails: if rewritten query diverges too far from the original intent, keep the original.

4. **Retrieval Agent**
   - Runs eval-approved `chunk_dense @cf/baai/bge-m3` first.
   - Optionally adds lexical/entity hybrid recall, hard-negative-aware overfetch, and rerank.
   - Emits trace rows for each subquery and strategy variant.

5. **Critique Agent**
   - Checks whether retrieved context is sufficient, relevant, non-conflicting, and answerable.
   - If not sufficient, allows one bounded re-retrieval loop with a revised query.
   - If still insufficient, forces an explicit "not enough evidence" answer instead of guessing.

6. **Generation Agent**
   - Generates only from injected context.
   - Must cite/source the supporting articles or chunks in trace metadata.

Agentic guardrails:

- Maximum two retrieval rounds per user question.
- Maximum three subqueries per plan.
- Hard timeout budget before falling back to linear `chunk_dense`.
- No autonomous external web browsing in production without a separate source-trust policy.
- Every agent decision must write trace metadata: `plan_id`, `step_id`, `subquery`, `strategy`, `critique_score`, `retry_reason`, and `stop_reason`.

Agentic eval gates:

| Metric | Gate |
|---|---:|
| Multi-hop Recall@10 | Must beat `chunk_dense` baseline on multi-hop cases. |
| MRR / NDCG@10 | Must not regress versus `chunk_dense` on entity-heavy cases. |
| Faithfulness | Must pass generation-side judge before production claim. |
| p95 latency | Must stay below 8000ms for agentic path or remain eval-only. |
| Loop safety | 0 unbounded loops in replay and production traces. |

Do not market Agentic RAG as shipped until this layer has its own eval slice. Resume-safe wording before implementation: "designed an Agentic RAG upgrade path with intent routing, query decomposition, critique, and bounded re-retrieval on top of an eval-gated chunk retriever."

## Eval Set Expansion Policy

The 21 approved cases are enough for an engineering gate, not enough for strong external claims. Grow the set to 30-50 human-approved cases before production rollout and to 50-100 before using the number as a resume headline without caveats.

AI-generated expansion is allowed only as candidate generation:

- Start from the 21 seed cases and generate paraphrases that preserve the same required evidence.
- Store synthetic/paraphrased cases with metadata such as `origin_case_id`, `synthetic_method`, `judge_model`, and `review_status = 'pending'`.
- Require human approval of both the rewritten query and the relevant evidence before including the case in official metrics.
- Keep canonical user-realistic cases separate from synthetic stress cases, so the leaderboard can show both `approved_real` and `approved_synthetic` slices.
- Never let synthetic cases replace source-diverse human cases from WeChat, Reddit, YouTube, official sources, GitHub, Product Hunt, RSS, and app usage logs.

```sql
-- 20260603 — Eval-only chunk dense retrieval RPC.
-- Does not change production answer-question retrieval.

create extension if not exists vector;
create extension if not exists pgcrypto;

create or replace function public.match_article_chunks_eval(
  query_embedding vector(1024),
  match_count integer default 30,
  chunking_version_filter text default null,
  chunk_overfetch_multiplier integer default 5,
  embedding_model_filter text default '@cf/baai/bge-m3'
)
returns table (
  chunk_id uuid,
  article_id uuid,
  title text,
  summary text,
  summary_en text,
  summary_zh text,
  article_content text,
  chunk_text text,
  chunk_index integer,
  chunk_rank integer,
  article_rank integer,
  score_dense double precision,
  embedding_source text,
  metadata jsonb
)
language sql
stable
security definer
set search_path = public
as $$
  with chunk_matches as (
    select
      c.id as chunk_id,
      c.article_id,
      c.chunk_text,
      c.chunk_index,
      c.chunking_version,
      c.token_estimate,
      c.language,
      1 - (c.embedding <=> query_embedding) as score_dense,
      row_number() over (order by c.embedding <=> query_embedding, c.id) as chunk_rank
    from public.article_chunks c
    where c.embedding is not null
      and (chunking_version_filter is null or c.chunking_version = chunking_version_filter)
      and c.embedding_model = embedding_model_filter
    order by c.embedding <=> query_embedding, c.id
    limit greatest(match_count * greatest(chunk_overfetch_multiplier, 1), match_count, 1)
  ),
  article_best as (
    select
      cm.*,
      row_number() over (
        partition by cm.article_id
        order by cm.score_dense desc, cm.chunk_rank asc
      ) as per_article_rank
    from chunk_matches cm
  ),
  deduped as (
    select
      ab.*,
      row_number() over (order by ab.score_dense desc, ab.chunk_rank asc) as article_rank
    from article_best ab
    where ab.per_article_rank = 1
  )
  select
    d.chunk_id,
    d.article_id,
    coalesce(n.title, n.title_zh, n.title_en, '') as title,
    n.summary,
    n.summary_en,
    n.summary_zh,
    n.article_content,
    d.chunk_text,
    d.chunk_index,
    d.chunk_rank::integer,
    d.article_rank::integer,
    d.score_dense,
    'chunk_dense_eval_v1'::text as embedding_source,
    jsonb_build_object(
      'chunking_version', d.chunking_version,
      'embedding_model', embedding_model_filter,
      'token_estimate', d.token_estimate,
      'language', d.language
    ) as metadata
  from deduped d
  join public.daily_news n on n.id = d.article_id
  order by d.article_rank asc
  limit greatest(match_count, 1);
$$;

revoke all on function public.match_article_chunks_eval(vector(1024), integer, text, integer, text) from public;
grant execute on function public.match_article_chunks_eval(vector(1024), integer, text, integer, text) to service_role;
```

export const BGE_EMBEDDING_MODEL = '@cf/baai/bge-m3'

export async function bgeEmbedSearchQuery(env, text) {
  const res = await fetchWithRetry(`${env.BGE_EMBEDDING_BASE_URL.replace(/\/$/, '')}/v1/embeddings`, {
    method: 'POST',
    headers: {
      Authorization: `Bearer ${env.BGE_EMBEDDING_API_KEY}`,
      'Content-Type': 'application/json',
    },
    body: JSON.stringify({
      model: BGE_EMBEDDING_MODEL,
      input: text,
    }),
  })
  if (!res.ok) throw new Error(`BGE embed ${res.status}: ${(await res.text()).slice(0, 500)}`)
  const json = await res.json()
  const embedding = json.data?.[0]?.embedding
  if (!Array.isArray(embedding)) throw new Error('BGE embedding response missing data[0].embedding')
  return embedding
}

    match_count: matchCount,
    chunking_version_filter: chunkingVersion,
    chunk_overfetch_multiplier: chunkOverfetchMultiplier,
    embedding_model_filter: embeddingModel,
  })
  return rows.map((row, index) => ({
    ...normalizeCandidate({
      id: row.article_id,
      title: row.title,
      summary: row.summary,

export function loadEntityLexicon(raw = process.env.RAG_EVAL_ENTITY_TERMS || '') {
  return String(raw)
    .split(',')
    .map(term => term.trim())
    .filter(term => term.length >= 2)

export function extractEntityTerms(question, lexicon = loadEntityLexicon()) {
  const text = String(question || '')
  const latinEntities = text.match(/\b[A-Z][A-Za-z0-9$.-]*(?:\s+[A-Z][A-Za-z0-9$.-]*){0,3}\b/g) || []
  const configuredTerms = lexicon.filter(term => text.toLowerCase().includes(term.toLowerCase()))
  const moneyTerms = text.match(/\$?\d+(?:\.\d+)?\s?(?:M|B|亿|万|万美元|亿美元)?/gi) || []
  const codeTerms = text.match(/\b[a-z]+[A-Z][A-Za-z0-9_-]*\b|\b[a-z]+(?:search|ctl|cli)\b/g) || []
  return [...new Set([...configuredTerms, ...latinEntities, ...moneyTerms, ...codeTerms])]
    .map(term => term.trim())
    .filter(term => term.length >= 2)
    .slice(0, 12)
}

export function expandRetrievalQueries(question, options = {}) {
  const original = String(question || '').trim()
  const entityTerms = extractEntityTerms(original, options.lexicon || loadEntityLexicon())
  const lexicalTerms = extractLexicalTerms(original)
  const entityQuery = [...entityTerms, ...lexicalTerms.slice(0, 6)].join(' ')
  return [...new Set([original, entityQuery].filter(Boolean))]
}

```js
export function fuseCandidatesWeightedRrf(candidateLists, weights = {}, k = 60) {
  const byId = new Map()
  for (const list of candidateLists) {
    for (const candidate of list) {
      const id = candidate.id || candidate.article_id
      if (!id) continue
      const source = candidate.metadata?.source_key || candidate.embedding_source || 'unknown'
      const sourceWeight = Number.isFinite(weights[source]) ? weights[source] : 1
      if (!byId.has(id)) {
        byId.set(id, {
          ...candidate,
          score_final: 0,
          metadata: { ...(candidate.metadata || {}), fusion_sources: [] },
        })
      }
      const existing = byId.get(id)
      existing.score_final += sourceWeight / (k + (candidate.rank || 1))
      existing.score_dense = existing.score_dense ?? candidate.score_dense ?? candidate.score
      existing.score_lexical = existing.score_lexical ?? candidate.score_lexical
      existing.metadata.fusion_sources.push(source)
    }
  }
  return [...byId.values()]
    .sort((a, b) => b.score_final - a.score_final)
    .map((candidate, index) => ({ ...candidate, rank: index + 1 }))
}
```

-- 20260603 — RAG metrics refinement diagnostics.
-- Read-only; use after dense/lexical/hybrid/chunk/entity replays.

-- 1. Latest strategy leaderboard with gate status.
with ranked_runs as (
  select
    r.*,
    row_number() over (
      partition by r.retrieval_strategy
      order by r.created_at desc
    ) as rn
  from public.rag_eval_runs r
),
latest as (
  select
    s.name,
    r.retrieval_strategy,
    r.retrieval_version,
    r.notes,
    m.total_cases,
    round(m.avg_recall_at_5::numeric, 3) as avg_recall_at_5,
    round(m.avg_recall_at_10::numeric, 3) as avg_recall_at_10,
    round(m.avg_mrr::numeric, 3) as avg_mrr,
    round(m.avg_ndcg_at_10::numeric, 3) as avg_ndcg_at_10,
    round(m.avg_hit_rate_at_5::numeric, 3) as avg_hit_rate_at_5,
    m.latency_p50_ms,
    m.latency_p95_ms,
    r.created_at
  from ranked_runs r
  join public.rag_eval_retrieval_metrics m on m.eval_run_id = r.id
  join public.rag_eval_sets s on s.id = r.eval_set_id
  where r.rn = 1
)
select
  *,
  case
    when avg_recall_at_5 >= 0.55
     and avg_recall_at_10 >= 0.70
     and avg_mrr >= 0.35
     and avg_ndcg_at_10 >= 0.55
     and avg_hit_rate_at_5 >= 0.55
     and latency_p50_ms <= 2500
     and latency_p95_ms <= 8000
    then 'passes_minimum_gate'
    else 'below_gate'
  end as strategy_gate_status
from latest
order by strategy_gate_status desc, avg_recall_at_10 desc, avg_mrr desc;

-- 2. Per-case best strategy movement.
with latest_runs as (
  select distinct on (retrieval_strategy)
    id,
    retrieval_strategy,
    created_at
  from public.rag_eval_runs
  order by retrieval_strategy, created_at desc
)
select
  c.question,
  lr.retrieval_strategy,
  cr.recall_at_5,
  cr.recall_at_10,
  cr.mrr,
  cr.ndcg_at_10,
  cr.hit_at_5
from latest_runs lr
join public.rag_eval_case_results cr on cr.eval_run_id = lr.id
join public.rag_eval_cases c on c.id = cr.case_id
order by c.created_at asc, cr.recall_at_10 desc, cr.mrr desc;

-- 3. Chunk-derived candidates that are gold evidence.
select
  r.retrieval_strategy,
  c.question,
  rc.rank,
  rc.candidate_type,
  rc.article_id,
  rc.chunk_id,
  rc.title,
  ge.relevance_grade,
  ge.review_status,
  rc.score_dense,
  rc.score_final,
  rc.metadata
from public.rag_eval_runs r
join public.rag_eval_case_results cr on cr.eval_run_id = r.id
join public.rag_eval_cases c on c.id = cr.case_id
join public.rag_retrieval_candidates rc on rc.retrieval_run_id = cr.retrieval_run_id
left join public.rag_eval_gold_evidence ge on ge.case_id = c.id and ge.article_id = rc.article_id
where r.retrieval_strategy like 'chunk_%'
order by r.created_at desc, c.created_at asc, rc.rank asc
limit 100;

export async function rerankCandidatesWithCloudflare(env, question, candidates, limit = 20) {
  const accountId = env.RAG_EVAL_CLOUDFLARE_ACCOUNT_ID
  const apiToken = env.RAG_EVAL_CLOUDFLARE_API_TOKEN
  const model = env.RAG_EVAL_CLOUDFLARE_RERANK_MODEL || '@cf/baai/bge-reranker-base'
  if (!accountId || !apiToken) {
    return await rerankCandidatesWithJudge(env, question, candidates, limit)
  }

  const contexts = candidates.slice(0, limit).map(candidate => ({
    text: [
      candidate.title || '',
      candidate.summary || '',
      candidate.chunk_text || candidate.excerpt || '',
    ].join('\n').slice(0, 4000),
  }))

  const res = await fetch(`https://api.cloudflare.com/client/v4/accounts/${accountId}/ai/run/${model}`, {
    method: 'POST',
    headers: {
      'Authorization': `Bearer ${apiToken}`,
      'Content-Type': 'application/json',
    },
    body: JSON.stringify({
      query: question,
      contexts,
      top_k: contexts.length,
    }),
  })
  if (!res.ok) {
    return await rerankCandidatesWithJudge(env, question, candidates, limit)
  }

  const data = await res.json()
  const rows = Array.isArray(data.result?.response)
    ? data.result.response
    : Array.isArray(data.result)
      ? data.result
      : []
  const scoreByIndex = new Map(rows.map(row => [
    Number(row.index),
    Number(row.score ?? row.relevance_score ?? row.value),
  ]))

  return candidates
    .map((candidate, index) => ({
      ...candidate,
      score_rerank: Number.isFinite(scoreByIndex.get(index))
        ? scoreByIndex.get(index)
        : Number.isFinite(scoreByIndex.get(index + 1))
          ? scoreByIndex.get(index + 1)
          : 0,
      metadata: { ...(candidate.metadata || {}), rerank_model: model },
    }))
    .sort((a, b) => (b.score_rerank ?? 0) - (a.score_rerank ?? 0))
}

export async function rerankCandidatesWithJudge(env, question, candidates, limit = 20) {
  const model = env.RAG_EVAL_RERANK_MODEL || 'qwen/qwen3.5-flash'
  const candidateText = candidates.slice(0, limit).map((candidate, index) => [
    `[${index + 1}] ${candidate.title}`,
    String(candidate.summary || '').slice(0, 1200),
  ].join('\n')).join('\n\n')

  const result = await callTokenRouterJson(env.TOKENROUTER_API_KEY, model, [
    {
      role: 'system',
      content: [
        'You are reranking retrieved news evidence for a RAG system.',
        'Return JSON only: {"scores":[{"index":1,"relevance_score":0-100,"reason":"short"}]}',
        'Score direct evidence highest. Penalize merely adjacent topic matches.',
      ].join('\n'),
    },
    {
      role: 'user',
      content: `Question:\n${question}\n\nCandidates:\n${candidateText}`,
    },
  ], Number(env.RAG_EVAL_RERANK_TIMEOUT_MS || 120000))

  const scores = Array.isArray(result.scores) ? result.scores : []
  const scoreByIndex = new Map(scores.map(row => [Number(row.index), Number(row.relevance_score)]))
  return candidates
    .map((candidate, index) => ({
      ...candidate,
      score_rerank: Number.isFinite(scoreByIndex.get(index + 1)) ? scoreByIndex.get(index + 1) / 100 : 0,
      score_final: Number.isFinite(scoreByIndex.get(index + 1)) ? scoreByIndex.get(index + 1) / 100 : candidate.score_final || 0,
      metadata: { ...(candidate.metadata || {}), rerank_model: model },
    }))
    .sort((a, b) => b.score_final - a.score_final)
    .map((candidate, index) => ({ ...candidate, rank: index + 1 }))

-- 20260603 — Human-curated RAG eval coverage candidate selection.
-- Read-only. Use these queries to choose eval candidates; do not auto-insert cases.

-- 1. Current coverage status.
select
  count(*) as total_cases,
  count(*) filter (where approved_relevant_gold > 0) as runnable_cases,
  count(*) filter (where approved_relevant_gold = 0) as cases_without_relevant_gold
from (
  select
    c.id,
    count(*) filter (
      where g.review_status = 'approved'
        and g.relevance_grade >= 2
    ) as approved_relevant_gold
  from public.rag_eval_cases c
  left join public.rag_eval_gold_evidence g on g.case_id = c.id
  group by c.id
) x;

-- 2. Production Q&A cases with weak retrieval traces or negative feedback.
select
  q.id as qa_log_id,
  q.article_id,
  dn.title,
  q.question,
  q.feedback,
  q.related_article_ids,
  r.candidate_count,
  r.injected_count,
  r.latency_ms,
  q.asked_at
from public.qa_logs q
join public.daily_news dn on dn.id = q.article_id
left join public.rag_retrieval_runs r on r.id = q.rag_retrieval_run_id
where q.asked_at > now() - interval '30 days'
  and (
    q.feedback = -1
    or coalesce(r.candidate_count, 0) = 0
    or coalesce(array_length(q.related_article_ids, 1), 0) = 0
  )
order by q.asked_at desc
limit 50;

-- 3. Recent source-diverse article candidates not already in eval cases.
select
  dn.id as article_id,
  dn.title,
  s.name as source_name,
  s.source_type,
  dn.created_at,
  left(coalesce(dn.summary_zh, dn.summary, dn.summary_en, dn.article_content, ''), 800) as evidence_preview
from public.daily_news dn
join public.sources s on s.id = dn.source_id
where dn.created_at > now() - interval '21 days'
  and not exists (
    select 1
    from public.rag_eval_cases c
    where c.primary_article_id = dn.id
  )
order by s.source_type, dn.created_at desc
limit 100;

-- 4. Existing cases that still need relevant approved evidence.
select
  c.id as case_id,
  c.primary_article_id,

from public.rag_eval_cases c
join public.daily_news dn on dn.id = c.primary_article_id

group by c.id, c.primary_article_id, dn.title, c.question

- The historical 9-case baselines are retained for retrospect, but current strategy decisions use the 21-case approved-gold baseline.
- Chunking and fusion address the current lexical-vs-dense evidence directly.
- Weighted sweeps must have unique strategy variants or diagnostics will collapse runs into one latest bucket.
- Rerank addresses the biggest metric gap: top-5/top-3 ranking.
- Embedding A/B is delayed and must use model-specific document embeddings because query-only model swaps are invalid.

If an agent is only building offline infrastructure and not interpreting metrics, Tasks 1, 2, and 2.5 may be implemented before Task 6 is complete. Do not choose winners, tune weights, approve rerank, run embedding A/B, or plan production rollout before Task 6 passes.
